## 1. Overview
Phase 7a is the foundation of Phase 7. It delivers the core certifications model, equipment competency tracking, cross-competency capture, audit-trail discipline, and layered event_job assignment gating. Onboarding (7b), external training (7c), and the Knowledge Base (7d) build on top.
Module name: neon_training. Manifest version: 17.0.7.0.0 (first release). Depends on: neon_jobs, neon_crm_extensions, neon_finance (for audit-trail pattern reuse). License: same as the rest of the Neon code.
Discipline carried from Phase 6:
Append-only on certification records (no perm_unlink for any group, per H3=A)
Corrections via new records with later effective_date — same as M5/M9 cost lines
ACL definitions in CSV with explicit per-group rows; ir.rule for record-level scope
Mid-build process: design pause → gate 1 approval → build → browser smoke → commit → gate 2

## 4. Key workflows
### 4.1 Certification acquisition (self-upload + admin verify)
Per B3=C. Two paths:
Path A: Ranganai records a cert directly for a crew member (admin-record-and-verify in one step). Common for internal equipment sign-offs.
Path B: Crew member self-uploads their certificate via portal (state=pending_verification). Admin (Robin / Ranganai / designated owner) reviews the attachment, confirms it's authentic, sets verified=True → state=active.

State machine:
draft → pending_verification → active → expired / suspended
'expired' happens automatically when date_expires passes (cron at midnight).
'suspended' is admin-driven (e.g. cert is revoked because original course turned out to be a fake).
### 4.2 Expiry tracking & reminders
Per B2=B (auto-expire with warning period) and B4=B (90/30/7 day reminders). Daily cron checks active certs:
90 days before expiry: WhatsApp + email + in-app TODO to the cert holder + Robin/Munashe + designated owner
30 days before: same channels, escalated tone ('renewal action needed within 30 days')
7 days before: final warning, daily TODO until expiry
On expiry date: state transitions active → expired, all assignment gates that depend on this cert recompute
### 4.3 Cross-competency capture
Per Robin's A4 note. After an event_job is closed (state=done), the system surfaces a TODO to the Lead Tech (Ranganai):
“Record any cross-competency demonstrations from this event. Was a crew member observed performing competently in a role outside their formal certifications?”
Ranganai logs cross-competency records as needed. These appear in 'Who can do X' queries with a different visual marker (cross-competency vs formal cert).
### 4.4 Layered assignment gating (the heart of 7a)
Per G2=D — three tiers, increasing severity. Each tier is its own milestone (M9/M10/M11).
Tier
When fires
Severity
Behavior
Tier 1: Assignment
Sales rep assigns user_id to role_line
Informational
Toast notification: '{User} lacks {cert(s)} for this role. Continue?' No block, but logged.
Tier 2: Quote acceptance
Quote moves to accepted state
Warning + override
If any role line has gate_status != qualified, the action_accept method shows a wizard requiring override_reason text before continuing. Reason is logged.
Tier 3: Event start
event_job.state → in_progress
Block + override
UserError unless an explicit override_reason has been recorded on the gate log within the prior 24h. Forces conscious decision-making at the critical moment.

Cross-competency records soften the gate: if a user has a relevant cross-competency record demonstrating competence in the missing area, the gate downgrades by one tier (block → warn, warn → info, info → pass). This makes Robin's A4 'data analytics' angle directly actionable.

No pre-existing training data to migrate — this is a greenfield module. Seeding at -u time:

## 10. Audit trail discipline
Following Phase 6's pattern (H3=A confirmed):
No perm_unlink on certification, cross_competency, assignment_gate_log models — for ANY group including training_admin
Corrections via new records with later effective_date (the existing record's state changes to 'suspended' with a notes entry; new record carries the corrected data)
All state transitions recorded in chatter (mail.thread on all main models)
create_uid / write_uid / create_date / write_date all preserved (Odoo default)

## 12. Open design questions
Items that need decisions during 7a build, not blockers but worth surfacing now:
Q1: Equipment model extension — do we extend the existing equipment model (from Phase 4 / Workshop) or create a new neon.training.equipment.requirement model that links? Cleaner separation favours the latter. Performance favours the former. Defer to M8.
Q2: Soft skill 'level' interpretation — e.g. 'Computer literacy: Basic / Standard / Expert' uses the same skill_level_mode as equipment competency. But 'Languages spoken' is more about presence/absence + comfort level. Some soft skills want a different model. Defer to M3 design pause.
Q3: Cross-competency → cert promotion — if Bob demonstrates MA3 competency 3 times via cross-competency records, should the system suggest creating a formal cert? Defer to M6 design pause (could be a future Phase 7.x feature).
Q4: Gate override visibility — should override-reason captures be surfaced in the Cash Flow Dashboard from Phase 6 (visibility for finance team), or only in the training_admin dashboard? Defer to M11 design pause.
Q5: Mobile UX for self-upload (B3=C) — crew use phones primarily. Should the portal be PWA-optimised? Defer to 7b (the portal lives there, not 7a).
